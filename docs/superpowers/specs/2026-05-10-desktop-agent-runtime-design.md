# 桌面 Agent Runtime 设计稿

**目标：** 构建一个以 macOS 为第一优先平台的桌面 Agent。宿主层使用 Swift + WKWebView，UI 使用 React；Agent Core 作为跨平台独立层运行。用户通过全局快捷键唤起，以用户主动提供的输入作为唯一初始上下文，由 LLM 按需调用 tool 读取环境信息、操作 App 或文件，并通过气泡展示后台执行进度与结果。

## 架构概览

系统拆分为几个边界清晰的部分：

- `触发/输入层`：全局快捷键、用户输入的 prompt、以及用户主动选中的文本。这些是唯一允许在任务开始时直接提交给模型的输入。
- `Agent Runtime`：负责创建任务会话、维护本次运行状态、驱动 LLM 与 tool 的循环经典的ReAct loop。
- `Tool 层`：所有针对桌面环境的读取与操作能力都暴露为统一 tool，只有当 LLM 决定调用时才会执行。
- `平台适配层`：第一版先实现 macOS，接口设计保持平台中立，后续可接入 Windows/Linux。
- `反馈层`：Agent 在后台运行时，通过气泡展示状态、过程消息与结果；未来可替换为桌宠而不影响 Runtime 核心设计。
- `LLM 接入层`：可通过 Vercel 提供的接口/SDK 连接模型，避免把 provider 逻辑写死在宿主里。

本设计的核心约束是：

`只有用户主动提供的内容会在初始阶段提交给 LLM。`

屏幕内容、窗口信息、剪贴板、文件内容、App 状态等世界状态一律不预加载，不默认注入模型，必须由 LLM 通过 tool 按需读取。这一点应当与 code agent 的 `read file` 行为保持一致。

## 第一版范围

- 全局快捷键唤起 `Spotlight` 风格的小输入框。
- 支持“先选中文本，再输入 prompt”的交互路径。
- 由 LLM 按需调用 tool 读取上下文，包括但不限于屏幕、窗口、OCR、剪贴板、文件、前台 App 信息。
- 由 LLM 按需调用 tool 执行操作，包括文件保存、文件编辑、App 操作等。
- Agent 后台运行时，使用气泡展示进度、状态与结果。
- 第一版只实现 macOS 宿主，但核心与平台接口要为未来多平台扩展留出清晰抽象。

## 第一版非目标

- 不实现复杂权限弹窗或细粒度授权交互。
- 不实现完整审计控制台。
- 不在第一版引入 tool 分类策略系统。
- 不实现桌宠形态 UI。
- 不实现云端调度或多端同步体系。
- 不把 Agent Core 绑死在 Swift 宿主里。

## 数据流

1. 用户按下全局快捷键。
2. 系统弹出输入框，收集用户 prompt；如果存在用户主动选中的文本，也一并作为输入。
3. Runtime 创建任务会话，并且只把用户主动提供的输入发送给 LLM。
4. LLM 判断是否需要额外上下文，如果需要，则调用相应 tool。
5. Tool 执行读取或操作，并把结果返回给 Runtime，再反馈给 LLM。
6. Runtime 持续输出状态消息到气泡，直到任务结束。

## 边界定义

- 快捷键、文本输入、用户选区属于用户触发的入口能力，不属于 tool。
- 上下文读取能力是 tool，由 LLM 决定是否调用。
- App 与文件操作能力也是 tool，由 LLM 决定是否调用。
- Runtime 负责会话状态、调度与循环控制，tool 只负责单次能力执行。

## 未来扩展点

- 在不改变agent Runtime 的前提下增加更多平台适配器。
- 在现有 tool 体系之上叠加权限系统与审计系统。
- 将当前气泡反馈层替换为桌宠形态，而不修改 Agent 主循环。
- 允许社区扩展并注册新的 tool，接入统一的 tool registry。
